# 3. NON-NEGOTIABLE DESIGN TRUTHS

## 3.1 Hero must stop carrying the whole show
The hero / emblem can remain important.
It cannot remain compositionally over-responsible.
If the whole frame does not change when something matters, the system is still underdeveloped.

## 3.2 Chamber is not background
The chamber must become:
- stage
- environment
- architecture
- lighting rig
- energy field
- framing machine
- spatial author
- dominant actor on cue

## 3.3 Before / during / after is mandatory
Every major event must visibly express:
- anticipation / gathering / pressure / compression
- impact / rupture / ignition / release / sweep
- aftermath / residue / ghost / stain / altered space / memory

## 3.4 Screen-space is a first-class system
Not decoration.
Not a default bloom bath.
Not a permanent haze.
It is a cue-authored language that should only spend itself when it means something.

## 3.5 Safe-tier is real product truth
`webgpu / safe` is the flagship production target unless explicitly disproven.
Design for it directly.
Do not build an expensive fantasy and degrade later.

## 3.6 Whole-frame authority beats local prettiness
Always ask:
- does this read across a room?
- does the whole frame respond when something matters?
- is the image still interesting with the hero reduced?
- is the world / chamber participating enough?
- are we spending brightness and glow intentionally?

## 3.7 Distinct cue classes must produce distinct image classes
A cue layer that is elegant in code but still makes mostly the same picture is not success.
The entire point is visible stage consequence.

## 3.8 More is not better
Not more brightness.
Not more particles.
Not more layers.
Not more motion.
Not more post.

Better means:
- more decisive framing
- more consequence
- stronger image classes
- more meaningful contrast between moments
- better pacing
- stronger memory of what happened

# 4. WHAT REAL WORLD-CLASS SHOWS ACTUALLY LOOK FOR

This section is here because the agent should not only think like a graphics programmer.
It needs to think like a show designer.

World-class reactive or music-driven visual systems tend to care about:

## 4.1 Phrase-scale timing, not just beat triggers
Big moments land on phrase turns, bar boundaries, structured buildups, withheld releases, and resets.
Not every beat deserves the same kind of spend.

## 4.2 Hierarchy of consequence
A tiny hat tick should not get the same kind of frame spend as a drop.
A drop should not feel like a slightly brighter beat.

## 4.3 Visual memory
Good shows remember what just happened.
Aftermath matters.
Residue matters.
Ghosts matter.
The space feels changed.

## 4.4 Stage-wide behavior
Real shows do not only animate a center object.
They use:
- stage architecture
- lasers
- fields
- sweeps
- framing changes
- blackout and reveal
- whole-stage escalations

## 4.5 Directionality
Energy does not only “pulse.”
It sweeps, converges, tears, opens, collapses, fans, rotates, cages, pours, and recoils.

## 4.6 Negative space and restraint
Darkness is not absence.
Black is a tool.
Silence is a tool.
Holding back is a tool.
Stillness between hits is a tool.

## 4.7 Across-room readability
Big-screen systems must be legible from distance.
That means:
- silhouette
- intentional brightness spend
- strong image classes
- clear massing
- not too much micro-detail dependence

## 4.8 Distinct roles inside the image
Different elements do different jobs.
Real shows separate:
- main structure
- accent lights
- beam / line systems
- atmospherics
- large-scale wipes
- transient hits
- background pressure
- hero/icon moments

## 4.9 Cue logic and operator logic
World-class systems usually have both:
- automatic / signal-driven behavior
- authored cue logic or operator-level control logic

VisuLive is headed toward the automatic/authored hybrid side.
The cue layer must evolve accordingly.

# 7. AUDIO / MUSIC-CONTROL CONTRACT — DO NOT GUESS

The agent must not assume the right numbers or feature families.
Start here, measure later, refine from evidence.

## 7.1 Design principle
Different visual systems should listen to different musical truths.

Do not let the whole show listen to:
- one FFT
- one amplitude value
- one beat trigger
- one BPM estimate

Instead separate:
- tempo
- phase
- impact
- presence
- pressure
- texture
- memory
- stability/confidence

## 7.2 Control families

### Tempo lane
Purpose:
- groove continuity
- beat-phase breathing
- non-hit motion continuity
- bar / phrase alignment

Signals:
- `bpm`
- `bpmConfidence`
- `beatPhase01`
- `beatIndex`
- `barIndex`
- `barPhase01`
- `phraseIndex`
- `phrasePhase01`
- `tempoStable`

### Impact lane
Purpose:
- rupture, flash, spend, emblem articulation, event triggers

Signals:
- `fullHit`
- `bassHit`
- `midHit`
- `highHit`
- `peakStrength`
- `peakIndex`
- `dropConfidence`
- `transientDensity`

### Presence lane
Purpose:
- environmental pressure, sustained participation, occupancy

Signals:
- `fullPresence`
- `bassPresence`
- `midPresence`
- `highPresence`
- `rms`
- `crestFactor` (optional)
- `energyDensity`

### Texture lane
Purpose:
- air, chatter, shimmer, agitation, sparkle, harshness handling

Signals:
- `airEnergy`
- `sparkleEnergy`
- `bandImbalance`
- `highActivityDensity`
- `waveformShimmer`
- `noisiness`

### Memory lane
Purpose:
- residue, delayed wake, phrase traces, aftermath, ghost shaping

Signals:
- `levelTrailWhole`
- `levelTrailBass`
- `levelTrailMid`
- `levelTrailHigh`
- `releaseTail`
- `energyAccumulation`
- `phraseCarryover`

### Stability lane
Purpose:
- decide how much to trust tempo-driven behavior
- reduce confused lock when detection is unstable

Signals:
- `analysisConfidence`
- `tempoStable`
- `phraseStable`
- `sourceReliability`

## 7.3 Starting band families
These are design starting points, not sacred truths.

- `sub`: 20–60 Hz
- `kick`: 50–110 Hz
- `bassBody`: 90–220 Hz
- `lowMid`: 180–500 Hz
- `midPresence`: 500–2500 Hz
- `air`: 2500–8000 Hz
- `sparkle`: 8000–16000 Hz

Optional more granular set for testing:
- `subPressure`: 20–45 Hz
- `kickPunch`: 45–90 Hz
- `bassGrowl`: 90–180 Hz
- `body`: 180–350 Hz
- `presence`: 350–1200 Hz
- `clarity`: 1200–3500 Hz
- `air`: 3500–8000 Hz
- `sparkle`: 8000–16000 Hz

## 7.4 Starting smoothing / decay defaults
These are recommended starting defaults.
They must be documented and adjustable.

- hit attack: 0–15 ms equivalent
- hit decay: 60–140 ms
- bass presence smoothing: 180–450 ms
- mid/high presence smoothing: 120–260 ms
- bpm confidence smoothing: 500–1200 ms
- phrase confidence smoothing: 1–2 bars
- residue window: 180–700 ms depending on cue family
- chamber pressure smoothing: 120–350 ms
- high-band chatter smoothing: 30–90 ms
- energy accumulation half-life: 0.5–2.5 bars depending on mode

## 7.5 Routing matrix

### ChamberRig listens to
- bass presence
- RMS intensity
- phrase position
- stage dominance intent
- energy accumulation
- release tail
- low-mid body when needed for structural agitation

Chamber behaviors driven by these should include:
- pressure
- enclosure
- field density
- spatial sweep
- reveal / collapse / cage / opening
- world ownership

### HeroRig listens to
- bass hits
- mid hits
- impact window
- articulation cues
- structural emphasis
- occasional phrase punctuation

Hero behaviors driven by these should include:
- emblem articulation
- seam/rim emphasis
- contraction/expansion
- decisive emitted behavior
- structural flashes, not generic brightening

### EventRig listens to
- beat phase
- bar/phrase position
- transitions
- cue envelopes
- dominance intent

EventRig behaviors should include:
- timing of before/during/after
- pacing
- inter-beat choreography
- motion envelope sequencing

### CompositorRig listens to
- impact lane
- residue window
- release tail
- transition class
- stain/ghost policy
- transformation intent

Compositor behaviors should include:
- selective afterimage
- impact flashes / wipes
- frame memory
- cue-shaped stress or tearing
- scene-state residue

### FramingRig listens to
- cue class
- stage dominance intent
- phrase position
- energy accumulation
- chamber ownership

Framing behaviors should include:
- image class shifts
- center-bias reduction
- negative-space opens/closes
- “hero demotion” on chamber/world-dominant cues

## 7.6 BPM and phrase use rules
BPM should not be used as a crude global trigger.

Use BPM / beat phase for:
- breathing motion
- groove-continuous drift
- rotational cadence
- pacing and oscillation timing
- phrase/grid alignment

Do not use BPM as the sole driver for:
- major cue class changes
- all flashes
- all chamber moves
- all emblem articulation

Those should depend on combined cue logic.

## 7.7 Confidence gating rules
When `bpmConfidence` or `tempoStable` is low:
- reduce strict beat-locking
- reduce phrase-dependent hard transitions
- lean more on presence, RMS, and transient logic
- allow freer environmental behavior

When confidence is high:
- strengthen groove lock
- align chamber pulses and sweeps more tightly to phrasing
- allow more precise cue spends

# 8. CUE CONSEQUENCE MAP — DO NOT LEAVE THIS ABSTRACT

The cue layer must not be merely names.
It must define visible world behavior.

## 8.1 Cue families

### Gather / Build
Before:
- pressure rises
- chamber tightens
- framing narrows or holds tension
- subtle residue accumulates

During:
- motion compresses or syncs harder
- environment densifies
- emblem prepares, but does not overspend

After:
- either breaks into rupture/release
- or sustains pressure if phrase turn is not reached

### Rupture / Drop / Impact
Before:
- visible pre-impact gathering
- reduced noise, more compression
- chamber or frame prepares directionally

During:
- entire frame changes
- chamber, compositor, and hero all react with role separation
- significant image-class shift is allowed

After:
- residue / flash memory / ghost / altered chamber state
- not just immediate reset

### Release / Bloom / Exhale
Before:
- high tension or post-impact recoil

During:
- openness, expansion, relief, dissolved pressure, lifting motion
- chamber may soften or unfold

After:
- lingering tone or quiet trail

### Isolate / Break / Thin
Before:
- system may subtract rather than add

During:
- carve space
- darken environment
- reduce structure to a more singular or brittle image

After:
- hold negative space / vulnerability / tension

### Haunt / Linger / Aftershock
Before:
- low immediate activity but meaningful memory window

During:
- ghosts, stains, frame scars, chamber echo, subtle environmental motion

After:
- not full reset; this cue class can inform the next state

### Reveal / Transition / Reset
Before:
- ambiguity or suspended tension

During:
- reconfiguration, opening, wipe, world shift, framing shift

After:
- establishes new phase / new image class / new chamber mode

## 8.2 Cue output fields to explicitly carry
At minimum:
- `cueClass`
- `cueFamily`
- `intensity01`
- `spendBudget01`
- `heroWeight01`
- `chamberWeight01`
- `compositorWeight01`
- `framingWeight01`
- `dominanceIntent`
- `preWindowMs`
- `impactWindowMs`
- `afterWindowMs`
- `transformationIntent`
- `residuePolicy`
- `blackoutPermission`
- `worldTakeoverPermission`

## 8.3 Important rule
Some cues should be subtractive.
Not every cue should add light, bloom, or motion.

Examples:
- pre-drop compression may reduce motion and brightness
- isolate/break cues may carve darkness
- transition cues may clear the frame before rebuilding

# 10. SCREEN-SPACE / COMPOSITOR LANGUAGE

The compositor must become a language, not a garnish.

## Candidate systems
- impact flash
- directional wipe
- selective bloom spend
- cue-shaped afterimage
- frame stain / energy scar
- ghosts / delayed outlines / spectral residue
- chromatic strain / split under stress
- brief exposure lift under earned intensity
- subtractive blackout / carved absence
- edge burn / rim echo
- environmental residue, not just full-frame smear

## Rules
- selective, not always-on
- cue-driven, not constant
- supports staging, does not replace it
- should not hide weak composition
- should not create mush
- should be validated in safe-tier

## Example mapping ideas
- rupture: frame-wide directional shock + selective residue
- haunt: localized stain/ghost, not massive flash
- release: soft frame memory, lighter bloom, broader air
- isolate: subtractive or edge-only post, reduced fill

# 12. VALIDATION / PROOF PACK PROTOCOL

## Required scenarios
Run captures against multiple audio conditions, not one song type.

At minimum:
- steady 4-on-the-floor dance groove
- half-time hip-hop beat
- dense DnB / high-hat-heavy track
- ambient build with weak kick
- vocal-heavy midrange passage
- sparse percussive section
- silence / room tone / HVAC hum
- “false drop” or fake energy swell with no real phrase turn
- long sustained release tail

## Required outputs per major pass
- 3 strong stills
- 3 weak stills
- notes on why each is strong/weak
- cue class attached
- chamber dominance rating
- compositor behavior notes
- before/during/after readability notes
- across-room readability judgment
- safe-tier quality judgment

## Evidence questions
- Did the whole frame change or mostly the hero?
- Did the chamber take over when it should?
- Did the cue family create a distinct image class?
- Is aftermath meaningful or just blur?
- Does the scene still look premium in `webgpu / safe`?
- Is the system overreacting to noise or weak transients?

## Failure conditions
A pass should be considered weak if:
- most strong frames still depend on the hero
- cue classes are semantically different but look too similar
- chamber never truly dominates
- residue is mushy or unreadable
- safe-tier loses the intended authority
- the system looks “busy” but not more authored

# 13. ANTI-GOALS / FAILURE TRAPS

Do not:
- drift back into orb-polish-first work
- solve everything with bloom
- hide weak staging under haze
- create lots of tiny reactions instead of decisive image shifts
- let chamber remain decorative
- let cue classes remain visually too similar
- assume the right numbers without documenting them
- assume the right routing without writing a routing matrix
- defer safe-tier truth
- produce progress theater instead of proof
- increase complexity without increasing image authority
- mistake constant motion for life
- mistake high brightness for intensity
- mistake more layers for sophistication
