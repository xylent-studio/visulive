# VisuLive Frontier Brief — Operating Protocol + Frontier Expansion

## What is missing from the current manager instructions

### 1. A speed doctrine
The manager needs explicit permission to optimize for leverage, not ceremony.

Add these rules:
- Prefer **one high-leverage implementation wave** over many tiny “safe” waves.
- Tests should protect contracts and recent regressions, not expand into a parallel project.
- Do not write broad new test suites unless a new contract or bug class genuinely needs them.
- For visual work, a fresh capture pack and proof review can be more valuable than 20 additional unit tests.
- Use tests for deterministic logic, routing, and telemetry math.
- Use capture proof for framing, chamber authority, image diversity, and show consequence.
- Avoid doc churn. Update canon only when the real priority stack changes.

### 2. A “build the stage, not the spreadsheet” rule
The current flow risks over-investing in telemetry and not enough in visible consequence.

Add:
- Telemetry exists to support image authority, not replace it.
- Never expand telemetry without defining the design decision it is meant to unlock.
- Never expand controls unless a real stage behavior cannot be reached without them.
- Never add a new knob when an internal cue policy or stage regime would solve it better.

### 3. A hard split between proof work and exploration work
The manager should explicitly distinguish:
- **Proof wave**: structural fixes, telemetry, acceptance gates, benchmark recapture
- **Exploration wave**: new scene families, transition families, compositor families, camera/shot behaviors

Otherwise the project risks mixing benchmark correction and frontier invention in one muddy pass.

### 4. A “don’t stop at AFQR” rule
AFQR is the benchmark driver right now, but the agent should not let one track overdefine the product.

Add:
- AFQR is the current correction benchmark, not the final taste benchmark.
- After AFQR improves, validate against one contrasting track:
  - slower / heavier / darker
  - lighter / more airy / vocal
  - high-hat-dense / percussive
- Benchmark diversity matters because the project is trying to become a show system, not a one-track trick.

### 5. A stronger creative frontier brief
The manager still needs explicit license to invent beyond the current scene grammar.

Add:
- The current scene is not sacred.
- The current hero grammar is not sacred.
- The project may need multiple flagship-caliber image families, not one scene stretched across all music.
- If a second scene family or scene-state family is the right answer, propose it.
- If a transition engine or show-sequencing layer above the current act system is needed, propose it.
- If browser-safe 2D/3D hybrid compositing is the missing unlock, lean into it.

## Working method that wastes less time

### What the human should do
- choose benchmark tracks
- review stills and short clips
- judge “strong / weak / close / dead” honestly
- decide which directions feel truly premium

### What the strategic LLM should do
- diagnose failure patterns
- refine prompts and priority order
- translate visual goals into coding tasks and acceptance gates
- pressure-test agent claims
- preserve canon and prevent drift

### What the coding LLM should do
- implement structural changes
- add or adjust contracts, telemetry, policies, and scene behavior
- run targeted tests
- generate proof packs
- report deltas clearly

### Anti-waste rules
- No huge generic test expansions.
- No speculative refactors without a named leverage goal.
- No giant doc rewrites unless canon is actually wrong.
- No “look how much I changed” behavior.
- No broad scene-art churn before governance and benchmark evidence are in.
- No shipping code just because it is clever.
- No overfitting the benchmark without at least one contrast track after the correction wave.

## What else the agent should be explicitly thinking about

## A. Scene families / image families
Right now the project still risks behaving like one scene with variable moods.

Push the manager to think in **image families**.

Potential flagship families:

### 1. Emblem / Artifact
The current strongest family.
- iconic central/emergent emblem
- chamber as pressure field / rig / architecture
- dark-value discipline
- electric seams / rims / emitted structure

### 2. Chamber / World Takeover
The world dominates, hero secondary.
- enclosure
- lattice gates
- collapse wells
- aperture mechanics
- sweeping energy walls
- laser-architecture feel

### 3. Beam / Vector / Corridor
Less orb-like, more directional.
- vectors
- vanishing-point energy
- tunnel / corridor / lane / runway logic
- stronger stage-depth and forward motion
- useful for builds, flight, propulsion, attack, reveal

### 4. Rift / Tear / Scar
Screen-space and world-space rupture.
- slits, tears, fractures, occlusion scars
- negative-space violence
- cue-shaped screen-space cuts
- aftermath residue and damage memory

### 5. Field / Swarm / Storm
Less singular object, more systemic world motion.
- particle weather
- wire storms
- charge migration
- magnetic field behavior
- image reads driven by pressure and directional flow

### 6. Cathedral / Stage-Architecture
The world feels built and monumental.
- arches, ribs, vaults, frames, portals
- giant luminous structures
- more “show rig” and less “object in void”
- strong for drops, reveals, and slow dramatic passages

The manager should not necessarily build all of these now.
But it should decide whether the current scene is meant to absorb these as act families, or whether some deserve distinct scene-state systems.

---

## B. Transition families
This is probably underdeveloped.

The show should not only have different states.
It should have different **ways of changing states**.

Transition families to consider:
- wipe / sweep
- collapse inward
- burst outward
- iris open / iris close
- gate slide
- lattice assemble / disassemble
- blackout cut
- overexposed flash then reveal
- residue dissolve
- directional scan / tracing reveal
- fracture / shatter / regroup
- vector handoff (one system hands motion direction to another)

The manager should think of transitions as authored stage language, not just lerps between parameter sets.

---

## C. Camera / shot logic
This is more important than the repo currently treats it.

The camera is not just a framing math problem.
It is part of the show language.

Possible shot behaviors:
- held icon shot
- pressure drift
- impact recoil
- slow advance
- world-reveal pullback
- rift-side glance / oblique shot
- axis-lock rupture shot
- aftermath distance hold
- chamber-dominant wide
- emergency blackout reset shot

Even if the camera remains restrained, the manager should explicitly think in terms of shot language.

---

## D. Negative / subtractive event vocabulary
The project still leans too additive.

Real show systems often gain power from subtraction:
- sudden darkness
- silhouette only
- light killed everywhere except one vector
- chamber disappearing before reappearing bigger
- hero going dark while world ignites
- temporary visual vacuum before impact
- removal of lines / particles / glow to create contrast

Tell the manager:
- build subtractive cue behaviors, not just additive ones
- some of the best moments should come from withholding and carving away, not only adding brightness

---

## E. Color systems beyond palette switching
The system may still be treating color too much like palette selection.

Push toward:
- cue-conditioned color spend
- dominant/accent/support role separation
- temperature shifts by cue family
- color memory after impact
- limited-color holds for authority
- selective desaturation before violent events
- environmental color vs hero color tension
- black and near-black as part of the palette system

The manager should ask:
- when should color be withheld?
- when should only one register of the image be allowed to color-shift?
- when should the world and hero deliberately disagree chromatically?

---

## F. Rhythm beyond hits
The manager should think beyond “beats trigger things.”

Explicit musical truths to exploit:
- beat phase
- bar cadence
- phrase boundaries
- density changes
- syncopation / off-beat chatter
- sustained pressure vs transient attack
- silence and breakdowns
- release-tail meaning
- repeated motif recognition if feasible
- dropout / fakeout / pre-drop withholding

This is where stage timing gets more premium.

---

## G. Surface / material logic
The emblem/world may still lack enough meaningful material states.

Potential states:
- dark structural mass
- thin rim energy
- filament lattice
- shell ignition
- seam-only mode
- internal pressure glow
- vaporized / translucent mode
- scarred / burnt / overdriven mode
- field-only / silhouette-only mode

The manager should consider whether some cue families should switch material logic more decisively instead of only modulating the same material setup.

---

## H. Residue / memory families
Aftermath should not be one thing.

Possible residue families:
- ghost trail
- chamber stain
- spatial echo
- scar line
- delayed secondary bloom
- fading vector memory
- lingering aperture damage
- afterimage silhouette
- pressure haze that decays nonlinearly

The manager should make residue a vocabulary, not one pass.

---

## I. Event scales
Not every cue should spend at the same scale.

Define event scales such as:
- micro: detail only
- local: hero/chamber subsystem change
- frame: whole composition shift
- world: environment-dominant shift
- show: act / family / scene-level transformation

This helps prevent every strong moment from trying to be world-ending.

---

## J. A deeper stage logic layer may still be missing
The repo may still need a stronger layer between cue state and visible choreography.

Possibly:
- `ShotDirector`
- `TransitionDirector`
- `StageCadencePolicy`
- `SceneFamilyRouter`
- `ConsequencePlanner`

These are not mandatory names.
But the concept matters:
there may still be one missing abstraction that decides *how* a cue becomes a staged event.

## What to tell the manager about testing and validation

Use this doctrine:

### Testing should be narrow and protective
Use tests for:
- routing logic
- cue thresholds
- telemetry calculations
- range mapping
- deterministic regressions

### Validation should be visual and benchmark-driven
Use proof packs for:
- framing
- chamber authority
- image diversity
- transition quality
- aftermath readability
- world-dominance delivery

### Avoid over-testing in these areas
Do not spend major time on exhaustive unit tests for:
- visual style nuance
- exploratory scene behavior
- one-off temporary tuning experiments

Instead:
- capture
- compare
- keep the strong
- kill the weak

- Do not write broad tests unless a new contract needs protection.
- Do not create more docs than code unless canon is actually stale.
